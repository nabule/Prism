Memosima 使用手册

当前完成状态

已经完成并通过真实测试的范围：

  • Memos、Sidecar API、worker 的 Docker Compose 启动。
  • Memos webhook 接入、任务入队、后台处理和管理接口。
  • OpenAI-compatible LLM 整理，默认使用 。
  • AI 整理 memo 创建，并写入 Memos 原生 REFERENCE relation。
  • 内置 /admin/ui 调试管理页面，可查看任务、重试任务和审核候选标签。
  • 候选标签创建、审核、审核后 active 生效。
  • AI 可从正文自动生成 tag 建议；未审核的新 tag 只进入候选列表。
  • AI 提示词可配置；支持保存默认提示词，也支持任务重试时临时覆盖。
  • 待澄清评论和 waiting_user 任务状态。
  • .txt、.md 附件解析到 artifacts 表。

还没有完成：docx/xlsx/pdf/drawio/Mind Elixir 专用解析、向量索引、问答、自动文档、备份恢复。

1. 准备配置

复制环境变量示例并填写真实值。不要把真实 .env 提交到 git。

cp .env.example .env
变量说明
SIDECAR_ADMIN_TOKENSidecar 管理接口 token。
MEMOS_API_TOKENMemos Personal Access Token，建议用 PAT，不用短期登录 token。
MEMOS_WEBHOOK_URLMemos 可访问的公网 Sidecar webhook URL。
API key，当前默认 LLM provider 使用。
OPENROUTER_API_KEYOpenRouter API key，作为备选 provider 使用。

2. 启动服务

所有常用命令通过 Nx 入口执行。

npx nx run sidecar:up

检查健康状态：

curl http://localhost:8080/health

3. 打开调试管理页面

浏览器打开：

http://localhost:8080/admin/ui

页面用于调试 Sidecar：查看健康状态、查询任务、筛选任务状态、重试 failed 或 waiting_user 任务、查看候选标签并执行通过/拒绝，以及编辑默认 AI 提示词。页面不会从服务器端读取 token；需要手动输入 SIDECAR_ADMIN_TOKEN，保存后只写入当前浏览器的 localStorage。

4. 创建或更新 Memos PAT

可以用探针创建 PAT，并写到临时文件。命令输出默认不会显示 token 原文。

npx nx run sidecar:probe-memos -- \
  --bootstrap-username test \
  --bootstrap-password testtest \
  --create-pat \
  --pat-output-env-file /tmp/memosima-pat.env

把临时文件中的 MEMOS_API_TOKEN 写入本机受控的 .env 或部署密钥管理位置。

5. 配置内容入口

本地部署默认使用 poll 模式，Sidecar 会主动轮询 Memos 最近 memo，不需要公网 webhook，也不需要在 Memos Web 中配置内网回调地址。

如果要使用 Memos 内置 webhook，Memos 0.28.0 会拒绝 webhook URL 指向 localhost、Docker 内网或私有 IP，需要公网 URL，或者用公网隧道转发到 http://localhost:8080。

npx nx run sidecar:probe-memos -- \
  --bootstrap-username test \
  --bootstrap-password testtest \
  --configure-webhook-url "$MEMOS_WEBHOOK_URL"

6. 处理一条 memo

在 Memos 中创建内容明确的 memo。默认轮询模式下，worker 会在短时间内发现新 memo，并创建 process_memo 任务；随后读取 memo、生成整理计划、创建 AI 整理 memo 并建立 relation。

查询任务：

curl -H "Authorization: Bearer<空格>$SIDECAR_ADMIN_TOKEN" \
  "http://localhost:8080/admin/jobs"

7. AI 自动生成 tag

配置 后，worker 会请求默认模型从 memo 正文中识别标签。AI 返回的 active_tags 只有已存在于正式标签集合时才会被直接使用；AI 发现的新标签会写入 candidate_tags 并进入审核列表，不会自动激活。

默认模型连接参数来自 config/models.yaml。当前默认 provider 为，base_url、default_model、temperature、max_tokens、response_format 和厂商扩展 extra_body 都可在该文件修改；真实 key 只放在本地 .env 或部署环境变量中，不提交到 git。

正式标签集合来自 config/taxonomy.yaml 中的 active 业务标签，以及已经通过审核写入 SQLite business_tags 表的标签。没有配置模型 key 时，系统仍会保留原有的显式 #tag 本地识别和模板整理。

默认提示词来自 config/prompts.yaml。在调试管理页面的“默认提示词”区域点击“保存默认”会写回该文件；点击任务“重试”时选择临时修改，只会把提示词写入当前任务 payload 的 llm_prompt_override，不会修改默认文件。提示词可使用 {active_tags}、{local_plan_json}、{content} 占位符。

8. 候选标签审核

memo 中出现未审核业务标签，或 AI 从正文提出新业务标签时，会进入候选标签列表。可以在 /admin/ui 中审核，也可以调用 API。

curl -H "Authorization: Bearer<空格>$SIDECAR_ADMIN_TOKEN" \
  "http://localhost:8080/admin/tag-candidates"

批准候选标签：

curl -X POST -H "Authorization: Bearer<空格>$SIDECAR_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"note":"纳入正式标签"}' \
  "http://localhost:8080/admin/tag-candidates/1/approve"

批准后标签会写入 Sidecar 的 active 标签集合，后续 worker 会直接使用。

9. 待澄清任务

短内容或包含明显疑问的 memo 会触发澄清评论，任务进入 waiting_user。补充内容后可重试：

curl -X POST -H "Authorization: Bearer<空格>$SIDECAR_ADMIN_TOKEN" \
  "http://localhost:8080/admin/jobs/1/retry"

10. 文本附件解析

上传 .txt 或 .md 附件后，worker 会下载并解析为 Markdown artifact。解析结果记录在 job result.attachments，正文保存在 Sidecar SQLite 的 artifacts 表。

11. 验证和构建

npx nx test sidecar
npx nx run sidecar:compile
npx nx build sidecar
