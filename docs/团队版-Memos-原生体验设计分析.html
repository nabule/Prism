团队版 Memos 原生体验设计分析

📦 归档说明：本文档分析的是「能否让团队版与个人版一样，把写/读体验放在 Memos UI 内」。经讨论决定暂不实施，团队版功能保持当前形态（独立 /teams/ui 入口 + Sidecar 自建 team_entries 表）。本文存档供后续如再次评估时直接复用，避免重新讨论同一架构问题。

1. 需求对齐

目标态对齐如下：

能力个人版当前团队版目标
写入笔记用户在 Memos UI 写 memo团队成员在 Memos UI 写 memo
阅读 AI 整理Memos UI 里读原 memo + AI 整理 memo（通过 REFERENCE 关联展示）团队成员在 Memos UI 里读自己/队友的原 memo + AI 整理 memo
管理管理员在 /admin/ui 管理任务/标签/提醒/备份管理员在 /admin/ui 管理团队 + 任务/标签/提醒/备份

等价于在问：能否让团队成员的写/读完全发生在 Memos UI 内？

2. 核心障碍：Memos 没有「团队」概念

必须先讲清这一点，否则所有方案都站不住：

  1. Memos 是单用户笔记应用：一个账号 = 一个独立笔记空间。没有 workspace / team / namespace 这类概念。
  2. 可见性只有 3 档：PRIVATE（仅 owner）/ PROTECTED（任何登录用户可见）/ PUBLIC（匿名可见）。没有「这条只给小组 A 看」。
  3. 没有 per-tag ACL：不能声明「#team/A/* 只对 A 组成员可见」。Memos 不知道「组」是什么。
  4. Sidecar 当前只有 1 个 Memos PAT：worker 以 host 用户身份轮询，所有 AI 整理 memo 都挂在 host 一个账号下。换团队 = 换身份，没有原生支持。

这就是当初团队版另起 team_entries 表 + 自建 /teams/ui 的根本原因 —— Memos 的数据模型撑不住「多团队 + 互相隔离」。

3. 四条可行路径

路径 A：标签前缀 + 共享单一 Memos（最便宜，但隔离名存实亡）

做法：所有人共用 host 这一个 Memos 账号，约定团队 X 的 memo 用 #team/X/... 标签。Sidecar worker 按标签前缀分流，AI 整理 memo 也带同前缀。

维度评价
Memos UX✅ 100% 原生
数据隔离❌ 假隔离 —— 同一个 Memos 账号下所有人都能看见全部 memo
成员账号不需要（共用一个）
改动量极小
适合场景完全没有隐私需求的家庭/小组（本质等于个人版换个名字）

结论：本质是「单账号开多个分类」，不是真团队。不推荐。

路径 B：每团队一个 Memos 容器（真隔离，资源换效果）

做法：Sidecar 通过 Docker API 编排 N 个 Memos 容器（memos-teamA:5230、memos-teamB:5230…）。每个团队独立 Memos 实例（独立 SQLite、独立账号体系）。Caddy 加 handle /m/{team-slug}/* → memos-{team-slug}:5230 反代。Sidecar worker 按团队 fan-out，每团队一份配置/PAT。

维度评价
Memos UX✅ 100% 原生
数据隔离✅ 真隔离（独立进程 + 独立 DB）
成员账号每个团队成员需要在该团队的 Memos 注册账号
AI 整理现成机制，按团队跑
改动量大 —— 容器编排、动态路由、worker 多实例化、UI 改动、备份脚本要改
资源占用每 Memos 容器 ~50–100MB RAM，10 团队 ≈ 1GB
适合规模个位数到几十个团队；不适合 SaaS 几千个团队
关键风险Sidecar「单 PAT 单 Memos」假设深入到 worker / models / backup 各处，需全部抽象化

结论：技术上最干净，工程量最大。本质是「多个个人版拼在一起」+ 团队管理皮，团队之间几乎零联动。

路径 C：保留 team_entries，把 /teams/ui 做成「Memos 同款体验」（推荐）

做法：数据模型不变（仍在 Sidecar team_entries 里），但把 /teams/ui 升级到 Memos 视觉/交互同款：

  • 写：Markdown 编辑器 + 拖拽附件 + #标签 自动补全 + 草稿保存（仿 Memos <MemoEditor>）
  • 时间线视图：卡片瀑布流，左侧原词条、右侧 AI 整理摘要并排（仿 Memos MemoView + REFERENCE 双向锚点）
  • AI 整理 entry（新功能）：worker 对 team_entries 也跑一遍现有整理流程
    • 触发：team 写入新词条 → 创建 process_team_entry 任务
    • 生成 AI 整理结果，存为 team_entries.kind=ai_summary，source_uid 指回原词条
    • 时间线把 ai_summary entries 折叠到原 entries 下方（视觉上与 Memos 个人版一致）
  • 附件解析：worker 现行 docparser（drawio / Mind Elixir / MinerU）原样复用
  • 提醒：reminders 现在从 memo 抽 #提醒 标签，扩到从 team_entries 抽
  • 管理后台：/admin/ui 的 TEAMS 区已有「团队/成员/邀请/词条」，再加「任务」「AI 整理日志」按团队过滤
维度评价
Memos UX🟡 视觉同款但是 Sidecar 自建（用户基本感知不出）
数据隔离✅ 真隔离（team_entries.team_id 外键）
成员账号不需要 Memos 账号；用现有 team token 体系
AI 整理✅ 复用现有 worker（process_memo → process_team_entry）
改动量中等 —— 前端编辑器与时间线视图（最大头）、worker 扩 1 类任务、提醒/解析复用
资源占用不增加容器
适合规模几十到几百个团队都没问题
关键风险编辑器/附件上传体验做到「Memos 同款」需要不少 UI 工作；部分用户可能仍坚持「真 Memos」

结论：投入产出比最高。核心思路是把「Memos UX」和「Memos 后端」解绑 —— 用户要的是写/读体验，不是 Memos 这个具体程序。

路径 D：B + C 的混合（极致体验，工程量也极致）

做法：每团队一个 Memos 容器（路径 B），同时 Sidecar 把团队全量 memo 镜像到 team_entries，跨团队检索 / 跨团队 QA 仍走 Sidecar 自建 UI。

维度评价
Memos UX✅ 写/读都用真 Memos
数据隔离✅ 真隔离
跨团队聚合✅ 通过 Sidecar 自建 UI
改动量路径 B + C 之和，最大
适合场景既要真 Memos 体验，又要 Sidecar 增值能力的高端用户

结论：技术上最优，工程上最重。除非确认需求强烈，否则先不做。

5. 推荐结论

主线推荐 = 路径 C（升级 /teams/ui 到 Memos 同款体验），路径 B 作为后续可选高级模式。

为什么 C 是主线

  1. 用户真正想要的是「像写 Memos 一样写、像看 Memos 一样看」，不是「用 Memos 这个程序」—— 二者可以解耦
  2. 团队隔离是硬需求，Memos 数据模型先天不支持。C 用 Sidecar 自己的表彻底解决
  3. AI 整理 / 附件解析 / 提醒三大现有能力可以直接复用，不需要 fan-out
  4. 不引入容器编排复杂度，备份脚本只需扩 team_entries 表
  5. 后续如果某些团队真的要「实体 Memos」，再叠加 B 做混合（路径 D）

为什么 B 不应该是主线

B 把「写/读 UX」绑死在「Memos 进程」上，反而牺牲了团队功能的灵活度：跨团队检索、超级 Prompt、向量 QA 都不在 Memos 里 —— 用户最终还是要在两个 UI 间切来切去。

6. 未决决策点

决定 C 之前需要先回答 3 个问题：

#问题影响
Q1 成员能不能接受「不是真 Memos UI，但视觉/交互同款」？ 直接决定 C 是否成立。如果用户死磕「必须是 Memos 那个程序」，只能走 B。
Q2 团队成员之间是否需要互相看到对方的「写作过程草稿」？ Memos 个人版没有，所以 C 也可以不做。若要做，编辑器需多人协作，差距大。
Q3 AI 整理结果要不要也同步一份到 Memos，让管理员能在 Memos 里浏览？ 加一份「写回 Memos host 账号、带 #team/{slug}/... 标签」的可选开关。低成本但偶尔有用。

7. 路径 C 的实施分期（若未来重启）

  1. 阶段 1：worker 加 process_team_entry 任务，复用现有 LLM 调用 / 标签治理 / 附件解析；AI 结果以 team_entries.kind=ai_summary 入库；/admin/ui 任务面板加团队过滤
  2. 阶段 2：/teams/ui 词条 tab 升级为 Memos 同款时间线 + Markdown 编辑器 + 标签自动补全
  3. 阶段 3：/teams/ui 加附件上传 + drawio / Mind Elixir / Office 解析（直接复用 docparser）
  4. 阶段 4：/teams/ui 加提醒抽取展示，对接团队级 Webhook（每团队一个 Bark / 钉钉）
  5. 阶段 5（可选）：把每团队 AI 整理同步回 Memos host 账号，做「管理员一站式浏览」入口

8. 触发重新评估的条件

出现下列任一情况，建议把本文档从存档里捞出来重新讨论：

  • 个人版用户开始大量反馈「能不能和家人/小组共享一部分笔记」
  • Memos 上游版本引入了原生的 workspace / 团队概念（解除核心障碍）
  • 项目从「自部署个人工具」转向「SaaS 多租户产品」
  • 有具体业务场景要求「同一个团队成员的写作过程实时协同」
